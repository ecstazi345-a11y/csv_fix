# Инструкция по заполнению Daily Progress (для мастеров и линейного ИТР)

> **Для кого:** мастера, бригадиры, линейный ИТР.  
> **Где вводить:** Airtable → таблица сменного отчёта (Daily Progress).  
> **Зачем читать:** одна ошибка в поле может «выпасть» из план-факта, EV и оплаты.

Связанные документы: [SYNC_REGULATION.md](SYNC_REGULATION.md), страница витрины **«Качество данных»**.

---

## 1. Зачем это нужно

Каждая смена попадает в общую систему:

**Airtable → Supabase → план-факт → EV → КС-6а / признание → деньги**

Если поля заполнены неточно, система не сможет сопоставить ваш факт с планом месяца. Тогда:

- объём «есть на площадке», но **не виден** в план-факте;
- **занижается** EV и производительность звена;
- **задерживается** приёмка и оплата.

---

## 2. Главное правило

**Если поле заполнено неправильно — факт может не попасть в план-факт, EV и КС-6а.**

План месяца уже заведён в Monthly Passport Plan. Сменный отчёт должен **совпадать по смыслу** с планом: тот же проект, объект, дисциплина, система, IWP, код BOQ.

---

## 3. Обязательные поля

### Project / Проект

| | |
|---|---|
| **Что означает** | Код или название стройки (например, объект заказчика). |
| **Как заполнять** | Выбрать из списка / писать **так же**, как в плане месяца. Без сокращений «от себя». |
| **Типичные ошибки** | Другой проект; опечатка; пустое поле. |
| **Последствия** | Факт не попадёт в сводку проекта; KPI по объекту будут неверны. |

---

### Month / Месяц

| | |
|---|---|
| **Что означает** | Месяц, к которому относится работа (формат как в системе: `2026-05` и т.п.). |
| **Как заполнять** | Месяц **фактического выполнения**, не дата создания записи. |
| **Типичные ошибки** | Прошлый/следующий месяц; пусто. |
| **Последствия** | Факт уйдёт в другой период; план-факт месяца «поплывёт». |

---

### Facility / Здание или объект

| | |
|---|---|
| **Что означает** | Корпус, здание, зона объекта (АБК, ЦТП, блок и т.д.). |
| **Как заполнять** | **Точно как в плане** — те же слова, без «АБК» / «абк» / «АБК-1» вперемешку. |
| **Типичные ошибки** | Другое написание того же здания; объект «на глаз». |
| **Последствия** | Ошибка **WRONG_FACILITY** — EV не сопоставится с планом (см. «Качество данных»). |

---

### Discipline / Дисциплина

| | |
|---|---|
| **Что означает** | Вид работ: ОВ, ВК, ЭОМ, СС, АПТ и т.д. |
| **Как заполнять** | Из справочника; **как в плане** на эту работу. |
| **Типичные ошибки** | Путаница ОВ/ВК; «общестрой» вместо дисциплины. |
| **Последствия** | **WRONG_DISCIPLINE** — факт не матчится с плановой строкой. |

---

### System / Система

| | |
|---|---|
| **Что означает** | Инженерная система внутри дисциплины (вентиляция, отопление, электрика…). |
| **Как заполнять** | Как в плане / BOQ; не придумывать новое название. |
| **Типичные ошибки** | Пусто; «разное»; другое имя той же системы. |
| **Последствия** | Разрыв ключа работы; факт **FACT_ONLY** (вне плана). |

---

### IWP / Рабочий пакет

| | |
|---|---|
| **Что означает** | Пакет работ (IWP) — связка объёма с планом и звеном. |
| **Как заполнять** | Код IWP **из плана месяца** на эту позицию. |
| **Типичные ошибки** | Пустой IWP; чужой IWP; опечатка в коде. |
| **Последствия** | Факт не привяжется к плану; звено «потеряет» объём. |

---

### BoQ_Code / Код позиции

| | |
|---|---|
| **Что означает** | Код сметной / плановой позиции (BOQ). |
| **Как заполнять** | Код из плана или BOQ; **не выдумывать**. |
| **Типичные ошибки** | Объём без кода; старый код; код другой работы. |
| **Последствия** | Невозможно посчитать EV и сопоставить с планом. |

---

### Quantity_Today / Объём за смену

| | |
|---|---|
| **Что означает** | Физический объём, выполненный за смену (шт, м, м², м³…). |
| **Как заполнять** | Число ≥ 0; единица измерения как в BOQ. Если работ не было — не «придумывать» объём. |
| **Типичные ошибки** | Пусто при работе; завышение; объём без BOQ. |
| **Последствия** | Искажение факта, EV и производительности. |

---

### Direct_Work_Hours / Прямые трудозатраты

| | |
|---|---|
| **Что означает** | Часы, когда бригада **реально работала** на объекте. |
| **Как заполнять** | Сумма по бригаде за смену; честно, без «красивых» цифр. |
| **Типичные ошибки** | 0 при работе; дубли с другой смены; путаница с простоем. |
| **Последствия** | Неверная производительность; ложные потери по звену. |

---

### Idle_Hours / Простои

| | |
|---|---|
| **Что означает** | Часы, когда бригада на объекте, но **не выполняла** работу по причинам простоя. |
| **Как заполнять** | Только если простой был; иначе **0**. |
| **Типичные ошибки** | Простой без причины; все часы в простой; скрыть простой в «работе». |
| **Последствия** | Искажение AC/CV, потерь и управленческих решений. |

---

### No_Work_Reason / Причина отсутствия работ

| | |
|---|---|
| **Что означает** | Почему не было работы или был простой (нет фронта, РД, материалов, погода…). |
| **Как заполнять** | Из списка причин; **обязательно**, если Idle_Hours > 0 или объём = 0 при явной смене. |
| **Типичные ошибки** | Пусто при простое; «прочее» без пояснения в поле комментария. |
| **Последствия** | Нельзя обосновать потери и допуск фронта; споры с заказчиком. |

---

## 4. Что нельзя делать

- Писать **один объект разными названиями** (АБК / Абк / АБК корпус).
- **Путать дисциплину** (ОВ вместо ВК).
- **Оставлять IWP пустым**, если работа по плану.
- Указывать **объём без BoQ-кода**.
- Ставить **простой без причины** (Idle без No_Work_Reason).
- Писать **комментарий вместо поля** — комментарий не заменяет Facility, IWP, BOQ.
- Копировать **чужую строку** смены без проверки полей.
- Менять **месяц «задним числом»** без согласования с ИТР.

---

## 5. Перед отправкой смены — чек-лист

- [ ] Выбран правильный **Project**
- [ ] Указан правильный **Month**
- [ ] **Facility** написан как в плане месяца
- [ ] **Discipline** совпадает с планом
- [ ] **System** совпадает с планом
- [ ] **IWP** заполнен и совпадает с планом
- [ ] **BoQ_Code** указан и верный
- [ ] **Quantity_Today** соответствует факту за смену
- [ ] **Direct_Work_Hours** отражают реальную работу
- [ ] Если был простой: **Idle_Hours** и **No_Work_Reason** заполнены

---

## 6. Почему это важно для оплаты

Цепочка для денег:

```
Daily Progress (Airtable)
    → синхронизация в Supabase
    → сопоставление с Monthly Passport Plan (план-факт)
    → расчёт EV и производительности
    → приёмка ПТО / КС-6а / признание заказчиком
    → оплата
```

**Ошибка в поле на смене** → факт не матчится с планом → **EV «висит» вне плана** → руководитель видит разрыв на витрине → **признание и оплата задерживаются**.

После смены диспетчер запускает синхронизацию (см. [SYNC_REGULATION.md](SYNC_REGULATION.md)). ИТР проверяет страницу **«Качество данных»** в Streamlit.
