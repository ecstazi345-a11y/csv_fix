# SMR Execution Logic — текущее решение

## 1. Текущий статус

Сейчас основная рабочая логика остаётся месячной:

Monthly Passport Plan → Daily Progress → Monthly Reconciliation → Streamlit

Недельный фильтр пока используется только для просмотра факта за выбранную неделю.

Важно: сейчас не выполняем жёсткое сопоставление плана и факта через `JOIN` по `week_key`, потому что в `Monthly_Passport_Plan` пока нет полноценного недельного плана.

## 2. Почему не делаем weekly JOIN сейчас

В `Monthly_Passport_Plan` поле `week_key` сейчас рассчитывается от `Plan_Work_Start_Date`.

Это показывает только предварительную неделю старта работ, но не означает, что весь плановый объём действительно запланирован именно на эту неделю.

Факт из `Daily Progress` может выполняться несколько недель:

- часть в W14;
- часть в W15;
- часть в W16.

Если делать `JOIN` по:

project_code + month_key + week_key + BOQ + IWP + System

то система может ошибочно считать часть факта как `FACT_ONLY`, хотя на уровне месяца эта работа входит в план.

Поэтому на текущем этапе:

- месячное сопоставление оставляем как основное;
- недельный фильтр показывает только факт недели;
- недельный план, недельное отклонение и недельный `PLAN_ONLY / FACT_ONLY / MATCHED` пока не считаем как управленческую истину.

## 3. Текущая правильная логика

Если выбран месяц:

- показываем месячный план;
- месячный факт;
- matched EV;
- fact only EV;
- plan only value;
- execution percent;
- chaos percent.

Если выбрана неделя:

- показываем факт выбранной недели;
- EV недели;
- direct work hours недели;
- idle hours недели;
- BOQ;
- IWP;
- System;
- Discipline;
- Facility / Building;
- Crew / Foreman, если доступны.

Неделя сейчас — это фильтр факта, а не полноценный контур недельного контроля.

## 4. Правильная будущая логика с июня

В июне необходимо построить полноценный недельный план внутри `Monthly_Passport_Plan`.

Тогда `Monthly_Passport_Plan` должен быть не просто месячным списком объёмов, а месячной производственной моделью, разложенной по неделям.

Одна строка плана должна описывать:

1. Месяц
2. Неделю
3. BOQ-код
4. Пакет работ / IWP
5. Систему
6. Дисциплину
7. Титул / здание / зону
8. Звено
9. Ответственного мастера
10. Плановый объём
11. Плановый МТР
12. Плановую стоимость работ
13. Плановую стоимость МТР
14. Плановые трудозатраты на объём в чел.-ч и чел.-мес
15. Плановые затраты труда в деньгах

Формула затрат труда:

Плановые затраты труда = Плановые трудозатраты × Ставка чел.-ч

## 5. Будущая сквозная цепочка

Правильная полная цепочка должна быть такой:

Executability → Acceptability → Monthly Passport Plan → Weekly Plan → Daily Progress → Plan/Fact Aggregation → Acceptance Packages → Inspection → Recognition → Cash

Смысл:

- `Executability` отвечает: можно ли физически выполнять работу.
- `Acceptability` отвечает: будет ли работа признана и оплачена.
- `Monthly Passport Plan` фиксирует, что именно планируется выполнить в месяце.
- `Weekly Plan` раскладывает месячный объём по неделям.
- `Daily Progress` фиксирует ежедневный факт.
- Aggregation сравнивает план и факт.
- Acceptance Packages собирают объёмы для ПТО, инспекции и последующего признания.

## 6. Будущая логика сопоставления

Когда недельный план будет настроен правильно, можно будет делать честный weekly reconciliation:

План недели → Факт недели → Отклонение → Причина → Ответственный → Влияние на деньги

JOIN тогда можно будет делать по:

project_code
month_key
week_key
BOQ
IWP
System
Discipline
Facility / Building
Crew

Только после этого weekly `MATCHED / FACT_ONLY / PLAN_ONLY` станет корректным.

## 7. Важное правило

Нельзя считать weekly plan/fact, если недельного плана нет.

Недельный фильтр без недельного плана = аналитика факта.

Недельный план + недельный факт = полноценный контроль исполнения.

## 8. Цель системы

Цель Execution-контура — не просто показать прогресс.

Цель — создать управляемую цепочку:

плановый объём → плановая стоимость → плановые трудозатраты → фактический объём → фактические трудозатраты → отклонения → признание → деньги

Только после этого можно корректно переходить к контуру ПТО, накопителю объёма, исполнительной документации, инспекции и приёмке.
